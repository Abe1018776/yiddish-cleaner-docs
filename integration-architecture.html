Integration Architecture

Consolidating three Yiddish language processing repositories into one unified system within yiddish-cleaner

1 Current State — Three Separate Systems

Each repository operates independently with its own tech stack, data model, and deployment. There is no shared database, no unified authentication, and no cross-system data flow.

2 Target Architecture — Unified System

Key Design Decisions

  • Python as sidecar, not rewrite: The alignment code uses stable-ts, faster-whisper, torch, and CUDA. Rewriting in PHP would be impractical. The Python service runs as a Docker container or subprocess, called by Laravel via HTTP.
  • Mapping logic moves to Laravel: The matching algorithm (date+keyword scoring) is simple enough to port to PHP. The frontend becomes a Vue/Inertia page with server-side filtering and pagination instead of client-side JSON.
  • Database is the single source of truth: The static data.json from yiddish-mapping is imported into the database. All mapping state persists server-side with user attribution, not in localStorage.
  • M2M pivot already exists: The audio_sample_transcription pivot table in yiddish-cleaner already supports many-to-many audio-transcript relationships, which is the foundation for mapping data.

Review

JEM reviews cleaned transcript with diff viewer + karaoke mode. Approves or requests re-clean. Validates transcription.

Classify

Benchmark samples flagged as gold standard. Training-ready samples added to training versions. ASR models tested.

6 Three Pipeline Visualization

Benchmark Pipeline

Gold standard audio samples used to evaluate ASR model accuracy. These are never used for training.

STEP 1
Select Gold Audio

Mark audio samples as is_benchmark=true. Must have validated base transcription.

STEP 2
Validate Transcript

Base transcription must be cleaned and validated. This becomes the reference truth.

STEP 3
Run ASR Models

Transcribe using Whisper, YiddishLabs, etc. Each creates an ASR transcription record.

STEP 4
Calculate WER

Standard WER + Custom WER (critical errors only). Word-level alignment enables karaoke diff.

STEP 5
Compare Models

Dashboard shows WER across models. Public benchmark page at /benchmark.

Training Pipeline

Prepare 50+ hours of clean, aligned audio-transcript pairs for ASR model fine-tuning.

STAGE 1
Needs Cleaning

Raw transcripts imported. Text_raw populated but text_clean is null. Status: pending.

STAGE 2
Cleaned / Waiting

Processors ran. Clean rate calculated. Diff available. Awaiting JEM review. Status: completed.

STAGE 3
Approved

JEM reviews diff, uses karaoke mode. Validates transcription (validated_at set). Audio sample status: review.

STAGE 4
Aligned

Word-level alignment via stable-ts. TranscriptionWord records created with timestamps.

STAGE 5
Training Version

Added to TrainingVersion with criteria. Export as dataset (audio + aligned text segments).

Post-Training / Publishing Pipeline

After a model is trained, test it on benchmarks, flag low-confidence outputs, and publish.

STEP 1
Load Model

Register new ASR model in system. Configure as provider in asr.php config.

STEP 2
Run Benchmarks

Transcribe all gold standard audio. Auto-calculate WER against validated reference.

STEP 3
Confidence Flag

Words below confidence threshold flagged. Segments with many low-confidence words surfaced.

STEP 4
Human Review

Review flagged words. Mark critical errors. Custom WER recalculated.

STEP 5
Publish

If benchmarks pass threshold, publish model. Update public benchmark leaderboard.

7 JEM ASR User Journey

End-to-end workflow from raw audio discovery to published ASR model, as experienced by the JEM team.

1

Mapping — Audio-Transcript Association

Browse the unified mapping interface. The system displays 4,669 audio files and 1,065 transcripts. Smart matching algorithm suggests pairs based on Hebrew date and content type keywords. Confidence scores (0-1) are color-coded green/amber. Accept suggestions or use full-screen search modal for manual matching. Many-to-many relationships supported. Singing and silence segments auto-detected during alignment.

Mapping/Index.vue MappingController MatchingService
2

Training Preparation — 50-Hour Selection + Cleaning Pipeline

Select audio-transcript pairs for the 50-hour training corpus. Each pair enters the cleaning pipeline: Needs Cleaning (raw text only) → Cleaned/Waiting (processors ran, awaiting review) → Approved (JEM validated) → Sent to Training (added to TrainingVersion). The review interface shows original vs. cleaned diff, karaoke-mode playback, and per-processor removal statistics. Clean rate scoring (0-100) prioritizes review queue.

CleanerService TranscriptionReview.vue DiffViewer.vue KaraokePlayer.vue
3

Benchmarking — Gold Standard Management

Designate specific audio samples as gold standard benchmarks (is_benchmark=true). These are strictly separated from training data. Manage the gold standard set via /benchmark/gold-standard. Run multiple ASR models against benchmarks, view WER comparison across models at /benchmark/compare. The system enforces that benchmark samples cannot be added to training versions.

BenchmarkController GoldStandard.vue Compare.vue
4

Transcript Versions — Multiple Outputs Per Audio

Each audio sample can have multiple transcription versions linked via the M2M pivot: the original manual base transcript, one or more ASR outputs (Whisper, YiddishLabs, etc.), and manually corrected versions. WER is calculated between any ASR version and the base reference. Diff comparison lets users see exactly what each ASR got wrong. Karaoke mode highlights words in real-time as audio plays, color-coded by confidence.

AudioSampleTranscription (pivot) WerCalculator TranscriptionWord
5

Post-Training Testing — Model Evaluation + Publish

After training a new model, register it in the system and run bulk transcription against the benchmark set. The system auto-calculates WER and surfaces words with low confidence for human review. Critical errors are flagged via the word review interface. Custom WER (counting only critical substitutions) gives a more accurate real-world error rate. If the model passes quality thresholds, it enters the publish flow and appears on the public benchmark leaderboard.

AsrManager AudioSampleTextReview.vue Benchmark/Model.vue
