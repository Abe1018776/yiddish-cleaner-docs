Matching Algorithm

BUILT Audio-Transcript Matching

The matching algorithm scores audio-transcript pairs on two axes:

  • Date matching: Year/month/day comparison scores 0.25–1.0
  • Keyword matching: Content type similarity ("Sicha", "Maamar", "Basi Lgani") boosts confidence
  • Combined range: 0.25–0.70 typical; user reviews via human-in-the-loop verification

Results: 1,809 matched pairs | 2,860 unmatched audio | 447 unmatched transcripts

Cleaning Pipeline

Eight sequential processors remove non-spoken content:

#ProcessorAction
1SpecialCharsRemove zero-width characters, normalize smart quotes
2SeifMarkerStrip Hebrew section markers
3TitleStyleRemove headings via style/heuristic detection
4BracketsEliminate inline [editorial] annotations
5EditorialHebrewRemove non-spoken Hebrew citations
6ParenthesesStrip (notes) and (inaudible) markers
7ForceRemoveApply configured regex patterns
8WhitespaceNormalize line breaks and spacing

Presets: minimal, titles_only, full_clean, with_editorial, heavy

Accuracy target: 99% — automated script output with change tracking.

LLM-based cleaning delegates to multiple providers with Yiddish-specific prompts:

  • OpenRouter — Multi-model gateway
  • Anthropic — Claude models
  • OpenAI — GPT models
  • Google AI — Gemini models
  • Groq — Fast inference

API credentials stored encrypted in ApiCredential model. Cleaning executes via CleanTranscriptionJob queue job.

Clean Rate Scoring (0–100 scale) evaluates removal quality:

RangeRatingMeaning
90–100ExcellentMinimal removals
75–89GoodReasonable removals
50–74ModerateSignificant removals
25–49LowHeavy removals — review recommended
0–24PoorExcessive removal — likely over-cleaned

Nine penalty rules apply; global penalties trigger if >50% text removed or <50 words remain.

03 Alignment Engine (stable-ts)

Core Capabilities

  • Word-level alignment using stable-ts library
  • Confidence scores per word and per segment
  • Karaoke mode data — timestamps + text sync for playback

  • Yiddish normalization — strips nikkud, cantillation marks, punctuation

The alignment engine runs as a Python subprocess invoked by Laravel queue jobs. The AlignTranscriptionJob sends audio + transcript and receives timestamped JSON.

WER Calculation Details

The WerCalculator applies Yiddish-specific normalization before computing metrics:

FormulaStandard WER = (Substitutions + Insertions + Deletions) / Reference Words

Custom WER  = Critical Errors Only / Reference Words
            (user marks which substitutions are truly wrong vs. acceptable variants)

Normalization steps: strip nikkud (Hebrew diacriticals), remove cantillation marks, remove punctuation, lowercase. This accounts for the orthographic variation inherent in Yiddish text.

Custom WER allows reviewers to mark substitutions as "acceptable orthographic variants" vs. "critical errors" — yielding a fairer metric for Yiddish where multiple valid spellings exist.

Regression Detection

How regression is detected across model versions

When a new model version is evaluated against the benchmark set:

  1. WER/CER are computed per audio file in the gold standard set.
  2. Per-file results are compared against the previous model version.
  3. Any file where WER increases by more than a configurable threshold flags a regression.
  4. Aggregate metrics (mean WER, median WER) are compared.
  5. The comparison page (/benchmark/compare) visualizes per-file deltas.

This ensures fine-tuning iterations do not degrade quality on specific audio characteristics (year, recording quality, speaker).

06 Transcript Version Management

Version Types

BUILT Manual (Base)

Original human-transcribed text from Word documents. Serves as the reference for WER calculation. Type: base.

BUILT ASR Hypothesis

Machine-generated transcription from Whisper, Gemini, or YiddishLabs. Type: asr. Includes word-level timestamps.

BUILT Cleaned

Post-cleaning version with non-spoken content removed. Stored as text_clean alongside text_raw.

TODO Fine-Tuned Model Output

Transcription from custom fine-tuned models. Will be stored as additional ASR versions with model provenance.

Diff Generation

BUILT DiffService

The DiffService generates line-level and word-level diffs using the Longest Common Subsequence (LCS) algorithm. Display uses red (removed) and green (kept) highlighting.

Any two transcript versions can be diffed: manual vs. ASR, raw vs. cleaned, Whisper vs. Gemini, etc.

Google Drive Integration

How source files are accessed from Google Drive

Audio and transcript source files are stored in Google Drive with shareable links. The system accesses them in two ways:

  • Batch Import: ProcessSheetBatchJob reads a Google Sheets URL containing file inventory, then downloads audio and extracts text from transcript documents (docx/doc/txt).
  • Direct Links: Each record stores a Google Drive shareable link for on-demand access. The mapping frontend uses these links for audio playback and transcript preview.
