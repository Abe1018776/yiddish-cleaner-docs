Yiddish ASR Benchmark

A full-stack web application for benchmarking Yiddish Automatic Speech Recognition models. Import audio + reference transcriptions, clean/normalize text, run ASR, calculate WER/CER, and compare models on a public leaderboard.

Laravel 12 Vue 3 Inertia.js v2 Tailwind CSS v4 SQLite

00 What This Project Does

End-to-end pipeline for evaluating how well speech-to-text models understand Yiddish

1

Import Audio + Transcripts

Bulk import audio samples and their human-written reference transcriptions from Google Sheets, file uploads, or manual entry.

2

Clean Reference Text

Yiddish religious texts contain editorial notes, Hebrew citations, section markers, brackets. The cleaning pipeline strips non-spoken content to create accurate ground truth.

3

Run ASR Models

Send audio to speech recognition APIs (YiddishLabs, OpenAI Whisper, local Whisper) and collect the hypothesis transcriptions with word-level timing data.

4

Calculate Accuracy

Compare ASR output to cleaned reference using WER (Word Error Rate) and CER (Character Error Rate) with Hebrew-aware normalization.

5

Public Leaderboard

Public benchmark page ranks ASR models by average WER/CER. Supports model comparison, per-sample drill-down, and a curated "gold standard" subset.

05 User Journey

Complete workflow from importing audio to publishing benchmark results

1

Register & Configure

User registers an account (with optional 2FA). Navigates to Settings → Credentials to add API keys for ASR providers (YiddishLabs, OpenAI) and LLM providers (Anthropic, OpenRouter, etc.). Connects Google account for Sheets/Drive access.

2

Import Audio + Reference Transcripts

Go to Import → Create. Paste a Google Sheet URL (rows = audio samples with links to audio files and transcript documents on Google Drive) or upload a spreadsheet file. The system queues batch import jobs that download each audio file and extract text from each transcript document (docx/doc/txt).

3

Monitor Import Progress

Import Runs page shows real-time progress (via WebSocket events). Each row shows status: pending → processing → completed/failed. Click into a run to see individual audio sample results.

4

Clean Reference Transcriptions

Navigate to a Transcription → Show page. The raw text from the document is displayed. Choose a cleaning method:
Rule-based: Select a preset (full_clean, heavy, etc.) that runs 8 text processors in sequence.
LLM-based: Select an LLM provider/model to intelligently remove non-spoken content.
A background job runs the cleaning and shows a diff view (red = removed, green = kept) when complete.

5

Review & Validate Cleaned Text

Inspect the diff view showing what was removed. Optionally edit the cleaned text manually. Check the Clean Rate score (0-100) indicating cleaning quality. Click "Validate" to approve the cleaned text as ground truth for benchmarking.

6

Run ASR Transcription

On an Audio Sample's Show page, select an ASR provider and model. Click "Transcribe". A background job sends the audio to the ASR API, receives the hypothesis text with word-level timestamps, and automatically calculates WER and CER by comparing against the validated reference text.

7

Word-Level Review (Custom WER)

Review word-by-word alignment between reference and hypothesis. For each substitution, mark whether it's a "critical error" (true mistake) or acceptable variant (Yiddish spelling differences). Custom WER = (insertions + deletions + critical_replacements) / total_reference_words. This gives a fairer accuracy metric for Yiddish.

8

View Public Benchmark

The public Benchmark page (no login required) shows a leaderboard ranking ASR models by average WER, CER, and Custom WER. Drill into any model to see per-sample scores. Use the Compare page to put two models side-by-side on shared audio samples. The Gold Standard page shows results on a curated high-quality subset.

06 Text Cleaning Pipeline

Transforms raw Yiddish transcriptions into clean ground truth by removing non-spoken content

Why Cleaning Is Needed

Yiddish religious lecture transcriptions (from Torah classes, Chassidic discourses) contain extensive non-spoken content: editorial brackets [ed. note], parenthetical notes (unclear), Hebrew scripture citations, section markers (seif aleph, etc.), document titles/headings, and special Unicode characters. None of this appears in spoken audio, so it must be removed before comparing against ASR output.

07 ASR Transcription & WER Calculation

Speech recognition drivers and accuracy measurement with Hebrew-aware normalization

ASR Drivers

DriverDescription
YiddishLabsDriver Async API: POST audio → poll status until completed → parse timestamps. Purpose-built for Yiddish.
WhisperDriver OpenAI Whisper API. Upload audio, get transcription + segments.
LocalWhisperDriver Runs Python Whisper locally via scripts/whisper_transcribe.py. No API key needed.

WER Calculation

WerCalculator uses Levenshtein distance with Yiddish-specific normalization:

  • Strips Hebrew nikkud (vowel marks) and cantillation marks
  • Strips punctuation
  • Lowercases text
  • Word-level WER = (S + I + D) / reference_words
  • Character-level CER = same formula on characters
  • Detailed error list: each error tagged as substitution, insertion, or deletion

Custom WER (Word Review)

Standard WER penalizes ALL substitutions equally. But in Yiddish, many "errors" are acceptable spelling variants. Custom WER lets users review each word and flag only truly wrong substitutions as "critical errors".

Custom WER = (insertions + deletions + critical_replacements) / total_reference_words

This gives a more meaningful accuracy metric that reflects real comprehension errors rather than orthographic variation.

10 Frontend Architecture

Vue 3 SPA with Inertia.js server-side routing, dark "Midnight Neon" design system

How Inertia.js Works

Inertia eliminates the need for a REST API. Instead:

  1. Laravel controller returns Inertia::render('Page', $props)
  2. First load: full HTML with embedded page data
  3. Subsequent navigations: XHR fetches only JSON props
  4. Vue component receives props and renders reactively
  5. Form submissions via router.post() with validation errors

Design System

"Midnight Neon Studio"

  • Dark background: #0a0a0f
  • Electric Azure: #00d4ff
  • Magenta: #ff006e
  • RTL support for Yiddish text
  • Inter + JetBrains Mono fonts
  • shadcn-vue component primitives (25+ components)

12 Queue Jobs

Background jobs handle all long-running operations: cleaning, ASR, importing

CleanTranscriptionJob

Runs text cleaning (rule-based or LLM). Updates transcription with cleaned text, clean rate, and metrics. Fires AudioSampleProcessed event.

TranscribeAudioSampleJob

Sends audio to ASR API, receives hypothesis text. Calculates WER/CER against reference. Stores word/segment timing data. Fires completion event.

ProcessSheetBatchJob

Reads Google Sheet rows, downloads audio from Drive, extracts text from transcript documents, creates AudioSample + Transcription records.

ProcessSpreadsheetFileJob

Same as sheet batch but from uploaded spreadsheet files instead of Google Sheets.

AlignTranscriptionJob

Runs WhisperX word alignment to get precise word-level timestamps for a transcription.

CalculateTranscriptionMetricsJob

Recalculates WER/CER/Custom WER metrics when reference text or word reviews change.

Real-time Updates

Background jobs fire Laravel events (AudioSampleProcessed, BatchCompleted, DocumentProcessed) that are broadcast via Laravel Reverb (WebSocket server). The Vue frontend listens via Laravel Echo and reactively updates the UI when jobs complete -- no page refresh needed.
